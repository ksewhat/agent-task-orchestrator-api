# Step 01. FastAPI 프로젝트 기본 구조 잡기

> 학습 일자: 2026-06-23
> 목표: FastAPI 기반 백엔드 서버의 기본 골격을 직접 만들어보고, 실제 서비스에서 쓰는 구조를 이해한다.

---

## 1. 이번 Step의 목표

LLM API나 DB를 붙이기 전에, 먼저 "서버 자체가 잘 돌아가는지"를 확인할 수 있는 가장 기본적인 구조를 만드는 것이 목표였다.

단순히 `main.py` 하나에 전부 몰아넣는 방식이 아니라, 앞으로 기능이 늘어나도 유지보수하기 쉬운 구조로 시작하는 것이 중요하다고 느꼈다.

---

## 2. 생성된 프로젝트 구조

```
agent-task-orchestrator-api/
├─ app/
│  ├─ __init__.py
│  ├─ main.py              ← FastAPI 앱 생성 및 라우터 등록
│  ├─ core/
│  │  ├─ __init__.py
│  │  └─ config.py         ← 환경변수 로딩
│  └─ api/
│     ├─ __init__.py
│     └─ routes/
│        ├─ __init__.py
│        └─ health.py      ← /health 엔드포인트
├─ .env.example
├─ requirements.txt
└─ README.md
```

처음에는 이 구조가 단순한 서버치고는 복잡해 보였다. 그런데 기능이 하나씩 추가될 때마다 "이 기능은 어디에 두면 되겠다"는 기준이 생기니까, 오히려 나중에 훨씬 편해진다는 걸 알게 됐다.

---

## 3. 각 파일의 역할

### `app/main.py`

FastAPI 앱 인스턴스를 만들고, 각 라우터를 연결하는 진입점 역할을 한다.

```python
app = FastAPI(title=settings.app_name, ...)
app.include_router(health.router, tags=["health"])
```

앱 설정과 라우터 등록이 이 파일 하나에서 관리된다. 여기서 무언가를 직접 구현하지 않고, 조립만 한다는 느낌이다.

### `app/api/routes/health.py`

`/health` 엔드포인트를 담당하는 라우터 파일이다. 기능 단위로 파일을 분리해두면, 나중에 `/tasks`, `/agents` 같은 새 기능을 추가할 때 파일만 하나 더 만들면 된다.

### `app/core/config.py`

환경변수를 로딩하고, 앱 전체에서 쓸 수 있는 `settings` 객체를 만드는 파일이다. `pydantic-settings`를 사용하면 타입 검증까지 자동으로 해줘서 편리하다.

```python
class Settings(BaseSettings):
    app_name: str = "Agent Task Orchestrator API"
    app_env: str = "local"
```

### `.env.example`

실제 비밀값은 `.env`에 넣고 git에는 올리지 않는다. 대신 `.env.example`을 커밋해두면, 팀원이 어떤 환경변수가 필요한지 바로 알 수 있다.

### `requirements.txt`

프로젝트에서 사용하는 패키지 목록이다. 이 파일 하나로 다른 환경에서도 `pip install -r requirements.txt` 한 줄로 동일한 환경을 만들 수 있다.

### `README.md`

프로젝트를 처음 받은 사람이 가장 먼저 보는 파일이다. 실행 방법만 잘 적어도 팀 협업에서 큰 도움이 된다.

---

## 4. FastAPI에서 라우터를 분리하는 이유

처음에는 `main.py`에 전부 몰아넣어도 된다고 생각했다. 하지만 기능이 10개, 20개가 되면 하나의 파일이 너무 길어지고, 어디를 수정해야 할지 찾기도 어려워진다.

라우터를 분리하면:
- 기능별로 파일이 나뉘어서 찾기 쉽다.
- 한 파일을 수정해도 다른 기능에 영향을 주지 않는다.
- 나중에 팀원이 생기면 역할 분담도 자연스럽게 된다.

FastAPI의 `APIRouter`는 이 구조를 만들기 위한 핵심 도구다.

---

## 5. `/health` API가 실제 서비스에서 가지는 의미

단순히 `{"status": "ok"}`를 리턴하는 API인데, 왜 만드는 걸까?

실제 운영 환경에서는 서버가 죽었는지 살아있는지를 자동으로 확인하는 시스템(모니터링, 로드밸런서, 쿠버네티스 등)이 이 `/health` 엔드포인트를 주기적으로 호출한다. 응답이 없으면 서버가 죽은 것으로 판단하고 재시작하거나 트래픽을 다른 서버로 넘긴다.

즉, 이 작은 API 하나가 서버의 "생존 신호" 역할을 한다.

---

## 6. `.env`와 환경변수를 사용하는 이유

API 키, DB 비밀번호 같은 민감한 정보를 코드에 직접 쓰면 안 된다. GitHub에 올라가는 순간 외부에 노출되기 때문이다.

`.env` 파일에 분리해두면:
- 코드에는 민감한 정보가 없다.
- 개발 / 스테이징 / 프로덕션 환경마다 다른 값을 쉽게 적용할 수 있다.
- `.env` 파일은 `.gitignore`에 추가해서 git에 올리지 않는다.

---

## 7. Uvicorn이 하는 역할

FastAPI는 웹 프레임워크고, 실제로 HTTP 요청을 받아주는 서버는 별도로 필요하다. Uvicorn이 그 역할을 한다.

```powershell
uvicorn app.main:app --reload
```

- `app.main` → `app` 패키지의 `main.py` 파일
- `app` → 그 파일 안의 `app` 변수 (FastAPI 인스턴스)
- `--reload` → 코드가 바뀌면 자동으로 서버를 재시작 (개발 중에만 사용)

Uvicorn은 ASGI 서버로, FastAPI의 비동기 처리 방식을 지원한다.

---

## 8. Swagger UI를 통해 API를 확인하는 의미

FastAPI는 코드만 작성하면 자동으로 `/docs`에 Swagger UI를 생성해준다.

- API 목록, 파라미터, 응답 형태를 시각적으로 확인할 수 있다.
- 별도의 Postman 설정 없이 브라우저에서 바로 테스트할 수 있다.
- 프론트엔드 개발자나 팀원에게 API 명세서 역할도 한다.

개발 중에 문서를 따로 작성하지 않아도 항상 최신 상태로 유지된다는 게 가장 큰 장점이라고 생각한다.

---

## 9. 이번 Step에서 배운 핵심 지식 포인트

| 개념 | 핵심 내용 |
|------|----------|
| FastAPI 앱 생성 | `FastAPI()` 인스턴스가 앱의 시작점 |
| 라우터 분리 | `APIRouter`로 기능별 파일 분리, `include_router`로 등록 |
| 환경변수 관리 | `pydantic-settings` + `.env` 파일 조합 |
| ASGI 서버 | Uvicorn이 HTTP 요청을 받아 FastAPI에 전달 |
| Health Check | 서버 생존 여부를 외부에서 확인하는 신호 |
| Swagger UI | 코드 기반 자동 API 문서 생성 |

## 10. 개인 학습 인사이트

**구조를 먼저 잡는 것이 왜 중요한지 이해했다.**
처음에는 "이 정도 기능에 이렇게 복잡한 구조가 필요한가?" 싶었다. 그런데 반대로 생각하면, 작을 때 구조를 잡는 게 훨씬 쉽다. 이미 코드가 커진 다음에 나누려면 훨씬 많은 수고가 든다.

**`/health` 하나도 운영 관점에서 보면 중요한 출발점이다.**
단순해 보이는 API지만, 실제 서비스에서는 이 엔드포인트가 없으면 로드밸런서나 모니터링 시스템이 서버 상태를 알 수 없다. 처음부터 챙겨두는 습관이 중요하다고 느꼈다.

**기능을 붙이기 전에 기반을 안정적으로 만드는 것이 먼저다.**
LLM API, DB, 인증 같은 기능들을 빨리 붙이고 싶은 마음이 있었다. 하지만 서버 구조 자체가 흔들리면 나중에 기능을 추가할수록 복잡도가 폭발적으로 늘어난다. 이번 Step에서 기반을 제대로 잡았으니, 다음 단계로 나아갈 준비가 된 것 같다.
